3. belenios

3.1. description

Belenios is a well-established e-voting protocol. It is based on Helios, a end-to-end verifiable e-voting protocol. In Helios, voters encrypt their vote to a quorum of trustee that will anonymise ballots using verifiable mixnets, and then publish the result. Every step is verifiable (mixnets, decryption).

Some contributions of Belenios are :

  • protocol formal verification
  • first class blank vote support
  • voters' eligibility verifiability
  • homomorphic aggregation (as an alternative for verifiable mixnets)
  • decentralized "thresold" trustee ceremony

3.2. similar systems

It is closely related to CHVOTE and losely related to ElectionGuard.

5.1. introduction

Protocols can be modelled

5.2. automated tools

These protocols are often complex and involve many parties. It follows that proofs are quite complex. Because each party can be either honest or dishonest, there is 2N scenarios to analyze for N parties. It would be very long to do every proof by hand, so we need automated tools such as ProVerif.

5.3. The symbolic (Dolev-Yao) model

When analysing cryptographic protocols, we need to chose between the symbolic model and the computational? model. It will model what primitives we use and what an adversery can do.

We will focus on the symbolic model, also called the Dolev-Yao model.

It's a model where cryptographic primitives are idealised. This includes, for exemple, the random oracle model, where hash functions are supposed to output truly uniform random values, while flaws could be found in practice [1]. This flaws would only be considered in the computational model.

It's a model that suppose a very strong attacker, called a Dolev-Yao attacker, that models a hostile network. This is practical because the Dolev-Yao attacker capture what can be done on a hostile internet. The attacker can intercept messages, send new messages from computable data (running any algorithm on known data), reorder messages, … (he is god on the network).

5.4. ProVerif

ProVerif est un outil de vérifcation automatique des protocoles cryptographiques dans le modèle symbolique (dit modèle de Dolev-Yao).

You model the protocol you want to verify using applied pi-calculus. This is then translated to Horn clauses, allowing explorationary techniques to take place.

In the formal model, you can use the rules to rewrite predicate (finding new horn clauses) from previously known.

You also model the properties you want to hold, also using Horn clause.

  • secrecy (modelled as not(attacker(secret)))
  • authentication
  • …

ProVerif can tell if this is true for all possibles traces, proving the property. Or can find a trace where it does not hold, finding (and showing) an attack.

5.4.1. ProVerif outputs

When verifying a protocol (a property), 4 output possibles:

  • satisfied: ?there is a trace that achieve this result? (and give such a trace, e.g. an attack)
  • not satisfied: ?there is no trace where the predicate hold?
  • not sure
  • not terminated

5.5. How to verify properties in ProVerif

5.5.1. Proof of secrecy (Confidentiality)

If the predicate attacker(M) form existing predicates, M is secret. (non-reachability of the secret) written: \(not attacker(secret)\) Remark: completness and soundness are delegated as property on the logical search done by the resolution prover
